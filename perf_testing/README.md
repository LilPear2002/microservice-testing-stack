# Phase 1：性能测试概念全景

---

## 1.1 性能测试是什么？

> 在**指定条件下**对系统施加**特定负载**，观测系统的**响应时间、吞吐量、资源利用率**，判断是否满足**性能需求**。

三要素：**负载 → 系统 → 指标**

```
┌──────────┐       ┌──────────┐       ┌──────────┐
│ 施压端    │ ────→ │  被测系统  │ ────→ │  指标采集  │
│ (Locust) │       │  (Web/API) │       │ (TPS/RT/…) │
└──────────┘       └──────────┘       └──────────┘
```

---

## 1.2 性能测试的 5 种类型

| 类型 | 英文 | 目的 | 典型问题 |
|------|------|------|---------|
| **负载测试** | Load Test | 验证系统在**预期负载**下是否满足指标 | 1000 并发下 RT < 500ms？ |
| **压力测试** | Stress Test | 找到系统的**性能拐点**和**极限** | 多少并发开始报错？CPU 100% 时 TPS 多少？ |
| **稳定性测试** | Soak/Endurance | 长时间运行是否**内存泄漏、资源不释放** | 跑 24 小时后 TPS 下降了吗？ |
| **尖峰测试** | Spike Test | 瞬间流量突增，系统能否**撑住不崩** | 秒杀场景：1 秒内并发从 100→10000 |
| **容量测试** | Capacity Test | 确定当前配置能支持**多少用户/数据量** | 单机 MySQL 能撑 5000 TPS 吗？ |

### 图示对比

```
负载测试：  ━━━━━━━━━━━━━━━━━━  (平稳)
压力测试：  ━━━━━━━━━━━━━━━━━━  ↗↗↗  (找到拐点)
稳定性测试：━━━━━━━━━━━━━━━━━━━━━━━━  (跑 8h/24h)
尖峰测试：  ━━━━━━╱╲━━━━━━╱╲━━━  (瞬间暴增)
容量测试：  固定负载 + 增大数据量  (100w 条数据 vs 1000w 条)
```

## 1.3 核心指标体系

```
                    ┌──────────────┐
                    │   性能指标    │
                    └──────┬───────┘
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │  吞吐量指标  │ │  响应时间指标 │ │  资源利用指标 │
    └─────────────┘ └─────────────┘ └─────────────┘
```

### 3.1 吞吐量 — 系统「多能扛」

| 指标 | 全称 | 含义 | 面试怎么说 |
|------|------|------|-----------|
| **TPS** | Transactions Per Second | 每秒处理的事务数 | 「压测时关注 TPS 峰值和拐点」 |
| **QPS** | Queries Per Second | 每秒查询数 | 「读多写少的场景 QPS 比 TPS 高」 |
| **HPS** | Hits Per Second | 每秒 HTTP 请求数 | 「一个页面可能有多个 HPS」 |
| **CPS** | Connections Per Second | 每秒新建连接数 | 「长连接 vs 短连接差异很大」 |

> **TPS vs QPS**：一个「下单」事务 = 查库存 + 扣款 + 写订单 = 3 个查询 → TPS=1, QPS=3

### 3.2 响应时间 — 系统「多快」

| 指标 | 含义 | 面试怎么说 |
|------|------|-----------|
| **RT / Avg** | 平均响应时间 | 「容易被长尾拉偏，不能只看平均」 |
| **P50** (中位数) | 50% 请求的响应时间 | 「一半用户感受到的延迟」 |
| **P95** | 95% 请求的响应时间 | 「只有 5% 的请求比这个慢」 |
| **P99** | 99% 请求的响应时间 | 「长尾优化目标，P99 稳定才算好」 |
| **Max** | 最大响应时间 | 「可能是 GC/锁竞争导致，需要排查」 |

```
示例：100 个请求的响应时间（ms）
  [10,10,10, ..., 10, 5000]  ← 99 个 10ms，1 个 5s
  
  Avg = 60ms    ← 看起来还行
  P99 = 5000ms  ← 真相：长尾很糟糕！

  面试金句：「我们不看平均，看 P95/P99 来评估用户体验」
```

### 3.3 资源利用率 — 系统「多累」

| 指标 | 关注点 |
|------|--------|
| **CPU** | 用户态/系统态/iowait 占比 |
| **内存** | 堆内存/非堆内存/是否持续增长（泄漏） |
| **磁盘 IO** | IOPS、吞吐量、await（排队时间） |
| **网络** | 带宽使用率、连接数、重传率 |
| **GC** | Full GC 频率和耗时（Java 系） |

### 3.4 业务指标 — 系统「多靠谱」

| 指标 | 含义 |
|------|------|
| **错误率** | 失败请求 / 总请求（目标 < 0.1%） |
| **成功率** | 1 - 错误率 |
| **超时率** | 超时请求占比 |

## 1.4 🔥 核心：功能测试 vs 性能测试流程对照

> 这是你最关心的问题——压测的「需求评审→用例→执行→Bug跟踪」怎么对应？

```
┌──────────────────────────────────────────────────────────────────┐
│                    功能测试流程                                   │
│                                                                  │
│  需求评审 → 用例编写 → 用例评审 → 执行 → Bug跟踪 → 回归          │
│     ↓           ↓          ↓        ↓       ↓        ↓          │
│  理解业务    设计场景    查漏补缺   跑用例   提单    修完再跑      │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
                              ⬇ 对应 ⬇
┌──────────────────────────────────────────────────────────────────┐
│                    性能测试流程                                   │
│                                                                  │
│  需求评审    →  场景建模  →  脚本开发  →  脚本Review              │
│  (定指标)       (压什么)     (怎么压)     (查漏补缺)              │
│                                                                  │
│  预压测      →  正式压测  →  监控分析  →  瓶颈定位                │
│  (基准线)       (阶梯加压)   (看指标)     (找根因)                │
│                                                                  │
│  调优        →  回归压测  →  报告输出                             │
│  (改代码/配置)   (验证效果)   (结论+建议)                         │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 详细对照表

| 步骤 | 功能测试 | 性能测试 | 产出物 |
|:---:|---------|---------|--------|
| **1** | 需求评审 | **需求评审（定指标）** | 性能需求文档：TPS≥1000, P99≤500ms |
| **2** | 用例编写 | **场景建模** | 压测场景：登录→浏览→下单 占比 1:5:2 |
| **3** | 用例评审 | **脚本 Review** | 检查 think_time、参数化、断言是否正确 |
| **4** | 测试执行 | **预压测（基准）** | 单机 10 并发基准线 |
| **5** | — | **正式压测（阶梯加压）** | 10→50→100→200→500 并发，记录每档指标 |
| **6** | Bug 跟踪 | **监控分析 + 瓶颈定位** | 找到 CPU 100% 或 DB 慢查询或连接池耗尽 |
| **7** | — | **调优** | 加索引 / 改连接池 / 加缓存 |
| **8** | 回归 | **回归压测** | 调优后再跑一次，对比指标 |
| **9** | 报告 | **报告输出** | TPS 峰值、拐点并发数、瓶颈原因、调优建议 |

### 关键区别

| 维度 | 功能测试 | 性能测试 |
|------|---------|---------|
| 关注点 | **对不对**（功能正确性） | **快不快、扛不扛得住**（非功能） |
| 结果 | Pass / Fail | 数值：TPS, P99, CPU% |
| 环境要求 | 功能环境 | **独立压测环境**（和生产 1:1 最好） |
| 数据要求 | 少量测试数据 | **铺底数据**（模拟生产数据量） |
| 执行频率 | 每次提测 | 大版本/大促前/架构变更 |
| 工具 | Postman/Selenium | JMeter/Locust/压测平台 |

## 1.5 压测环境要求

```
                    生产环境
                  ┌──────────┐
                  │ 8C16G×10 │  ← 理想：1:1 环境
                  └──────────┘
                       
                    vs
                       
                    压测环境
                  ┌──────────┐
                  │ 4C8G×2   │  ← 现实：缩配环境
                  └──────────┘
```

| 要求 | 说明 |
|------|------|
| **独立环境** | 不能和生产共用，避免影响真实用户 |
| **配置一致性** | 至少同架构（同 OS、同中间件版本） |
| **铺底数据** | 数据库要有生产量级的数据（不能是空库！） |
| **监控体系** | Prometheus + Grafana 或云厂商监控 |
| **隔离** | 网络隔离，压测流量不经过生产网关 |

## 1.6 压测执行策略：阶梯加压

```
TPS
  ▲
  │                          ╭─── 饱和点（加并发 TPS 不涨）
  │                     ╭────╯
  │                ╭────╯
  │           ╭────╯
  │      ╭────╯                     ← 线性增长区
  │ ╭────╯
  │╱                                ← 基准线
  └──────────────────────────────────→ 并发数
      10   50   100  200  300  500
      
  每档跑 5-10 分钟，记录 TPS、RT、CPU、错误率
  找到：最佳并发、最大并发、拐点并发
```

### 关键并发定义

| 名称 | 含义 | 如何判断 |
|------|------|---------|
| **最佳并发** | 性价比最高 | TPS/并发 比值开始下降前 |
| **最大并发** | 系统能承受的上限 | 错误率开始 > 1% 之前 |
| **拐点并发** | 性能开始劣化 | RT 突然飙升、TPS 不再线性增长 |

## 2.2 Locust 核心特性速查

| 特性 | 代码 | 说明 |
|------|------|------|
| **HttpUser** | `class EcommerceUser(HttpUser)` | Locust 用户基类，自带 HTTP client |
| **@task(weight)** | `@task(5)` | 权重比例：浏览:搜索:详情:购物车:下单 = 5:3:2:1:1 |
| **wait_time** | `wait_time = between(1, 3)` | 两次操作间等 1~3 秒（模拟真实用户） |

| **HTML 报告** | `--html reports/report.html` | 命令行自动生成 |

## 2.3 核心代码解析

### 2.3.1 主用户类 —— HttpUser

```python
class EcommerceUser(HttpUser):
    wait_time = between(1, 3)  # 每次操作后等 1~3 秒

    def on_start(self):
        """登录获取 Token"""
        resp = self.client.post("/api/login", json={...})
        self.token = resp.json()["token"]
        self.client.headers["Authorization"] = f"Bearer {self.token}"

    @task(5)  # weight=5，执行概率最高
    def browse_products(self):
        resp = self.client.get("/api/products", params={"page": 1})
        assert resp.status_code == 200

    @task(1)  # weight=1，执行概率最低
    def checkout(self):
        CheckoutFlow(self).run()  # 委托给顺序任务集
```

### 2.3.2 多步骤顺序任务 —— SequentialTaskSet

```python
class CheckoutFlow(SequentialTaskSet):
    def on_start(self):
        self.product_id = random.choice(PRODUCTS)  # 随机选一个商品

    @task
    def step1_browse(self):
        self.client.get("/api/products")  # 第 1 步：浏览

    @task
    def step2_detail(self):
        self.client.get(f"/api/products/{self.product_id}")  # 第 2 步：详情

    @task
    def step3_cart(self):
        self.client.post("/api/cart")  # 第 3 步：加购

    @task
    def step4_order(self):
        self.client.post("/api/orders")  # 第 4 步：下单
        self.interrupt()  # 流程结束，退回主任务循环
```

### 2.3.3 wait_time 策略

```python
# 方式 1：固定等待
wait_time = constant(2)          # 每次等 2 秒

# 方式 2：随机区间（推荐）
wait_time = between(1, 3)        # 1~3 秒随机

# 方式 3：按任务不同等待
def wait_time(self):
    if self.last_task == "checkout":
        return random.uniform(3, 8)  # 下单后休息久一点
    return random.uniform(1, 3)
```

## 2.5 ⚠️ 踩坑记录

### 坑 1：`with self.client` 需要 `catch_response=True`

```python
#  在 Locust 2.x 中会报错
with self.client.get("/api/x") as resp:
    assert resp.status_code == 200

# ✅ 直接调用，不用 with
resp = self.client.get("/api/x")
assert resp.status_code == 200

# ✅ 或者加上 catch_response=True
with self.client.get("/api/x", catch_response=True) as resp:
    if resp.status_code != 200:
        resp.failure("非 200")
```

### 坑 2：assert 会让虚拟用户「死掉」

```python
#  Locust 里 assert 失败 → 该虚拟用户直接崩溃
#  生产压测中应该捕获并标记为 failure，让用户继续跑
resp = self.client.get("/api/x")
if resp.status_code != 200:
    resp.failure(f"Unexpected status: {resp.status_code}")
```

### 坑 3：忘了 `host` 参数

```python
#  没设 host → 所有请求发到空地址
# ✅ 命令行加 --host
locust -f locustfile.py --host http://localhost:8080
```

### 📝 面试话术 — Locust

> **面试官**：「用过 Locust 吗？和 JMeter 比有什么优势？」
>
> **答**：「用过。Locust 的核心优势是**用 Python 代码定义压测场景**，不是 XML——这让我们可以做复杂的参数化、动态逻辑、if/else 分支。比如模拟真实流量模型：浏览 50%、搜索 30%、下单 10%，用 `@task(5)` / `@task(3)` / `@task(1)` 的 weight 就能实现。

## 3.1 为什么需要压测平台？

### 直接用 JMeter/Locust 有什么问题？

| 问题 | 表现 | 平台怎么解决 |
|------|------|-------------|
| **协作难** | 脚本散落在各人电脑上 | 平台统一管理，在线编辑/Git 同步 |
| **施压能力不足** | 单机 JMeter 最多几千并发 | 分布式施压机集群，横向扩展 |
| **监控割裂** | 压测工具看不到系统指标 | 集成 Prometheus/Grafana 实时大屏 |
| **报告分散** | 每次手动截图/复制 TPS | 自动生成报告，历史对比 |
| **环境混乱** | 压到线上、把生产打挂了 | 流量标识 + 染色 + 安全熔断 |
| **无法回归** | 每次手动跑，忘了上次的配置 | 定时任务 + 基线对比 + 自动告警 |
| **权限失控** | 谁都能发起压测 | 审批流 + 角色权限（只读/执行/管理） |

## 3.6 面试话术：压测平台

> **面试官**：「说说你们公司的压测平台是怎么设计的？」
>
> **答**：「我们有一套自研压测平台（或基于开源的二次开发），核心包含三层：
>
> **第一层是 Web 控制台**，提供脚本在线编辑、场景配置、任务创建、实时大屏和报告中心。支持 JMeter 和 Locust 两种引擎。
>
> **第二层是调度层**，负责任务分发和资源调度。用户提交压测后，Scheduler 会根据施压机池的负载情况，选择空闲的 N 台 Worker，下发脚本和加压策略。支持阶梯加压——比如 100→500→1000 并发，每档跑 5 分钟。
>
> **第三层是执行层**，施压机（Worker）上运行 Agent 进程接收任务，按策略动态调整并发数，实时上报 TPS、RT、错误率等指标给 Prometheus。
>
> 安全方面，我们做了流量染色——压测流量在 Header 上加标识，网关识别后路由到影子表，不污染生产数据。同时设了自动熔断：错误率超过 5% 或 P99 超过 3 秒，平台自动停止压测。
>
> 压测结束后，平台自动生成报告——包含各接口的 TPS 峰值、P95/P99、资源使用曲线、与历史基线的对比，并发送到企业微信群。」

## 4.4 Step 3：脚本开发 —— 怎么压

关键代码见 `src/staged_locustfile.py`：

```python
class StagedLoadShape(LoadTestShape):
    """自定义阶梯加压策略"""
    stages = [
        {"name": "预热",    "duration": 60, "users": 10,  "spawn_rate": 5},
        {"name": "基准",    "duration": 60, "users": 50,  "spawn_rate": 10},
        {"name": "中等负载", "duration": 60, "users": 100, "spawn_rate": 10},
        {"name": "高负载",   "duration": 60, "users": 200, "spawn_rate": 20},
        {"name": "极限测试", "duration": 60, "users": 300, "spawn_rate": 30},
    ]

    def tick(self):
        run_time = self.get_run_time()
        cum = 0
        for stage in self.stages:
            cum += stage["duration"]
            if run_time <= cum:
                return (stage["users"], stage["spawn_rate"])
        return None  # 压测结束
```

| **RPS 线性增长** | 并发 10→300，RPS 6→156 线性 | ✅ 好 |

## 4.7 Step 6：瓶颈定位

```
  分析 /api/orders 为什么慢？

  ┌─────────────────────────────────────────────┐
  │  /api/orders 代码逻辑（target_server.py）    │
  │                                             │
  │  simulate_db_query(80, 200)   ← 80~200ms   │
  │  simulate_slow_query()        ← 5% 100~500ms│
  │  simulate_slow_query()        ← 又一个！     │
  └─────────────────────────────────────────────┘

  原因：
  1. DB 查询耗时 80~200ms（基础延迟）
  2. 有 5% 概率触发慢查询 100~500ms
  3. 订单接口调了两次 simulate_slow_query()
     → 同时触发两个慢查询的概率更高
     → 造成 P99 = 620ms 的长尾
  
  对于 2C2G 单机，这个性能在预期范围内。
  如果是生产，优化方向：
  ├── 慢查询加索引
  ├── 减少 DB 查询次数（合并查询）
  └── 引入缓存（Redis）
```
