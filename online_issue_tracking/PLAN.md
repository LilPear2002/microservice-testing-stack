# 线上问题追踪与改进 — 实操计划

> **核心理念**：用一个小型 buggy API 项目，模拟一次完整的「故障 → 发现 → 定位 → 改进」闭环

---

## 阶段 ①：构建带 Bug 的演示应用

**目标**：搭建一个 Python FastAPI 服务，内置 3 类线上常见 Bug

| Bug 类型 | 具体表现 | 对应线上场景 |
|----------|---------|-------------|
| 间歇性 500 | 特定参数偶尔抛异常 | 未处理的边界条件 |
| 慢接口 | `/slow` 接口有时耗时 >3s | 数据库慢查询 / 外部依赖超时 |
| 内存泄漏 | 某个接口每次调用后不释放对象 | 大对象未回收 |

**做什么**：
- 用 FastAPI 写 4~5 个接口（含正常 + 3 个 buggy 接口）
- 集成 Prometheus 指标暴露（`prometheus_client`）
- 结构化日志（JSON 格式 + requestId）
- Docker 化，打包成镜像

**产出**：一个可观测的、有病的 API 服务 🐛

---

## 阶段 ②：部署到 k3s + 搭建监控

**目标**：在 k3s 上部署应用，并挂上 Prometheus 监控

**做什么**：
- 启动 k3s（上次已经装好了）
- 编写 Deployment + Service 的 YAML
- 部署 **轻量版 Prometheus**（只抓取本应用指标，内存限制 200MB）
- 配置 Prometheus 抓取应用的 `/metrics` 端点
- 在 Prometheus UI 里验证能看到指标

**资源预算**：
| 组件 | 预估内存 |
|------|---------|
| k3s（基础） | ~400MB |
| App x2 副本 | ~100MB |
| Prometheus | ~200MB |
| **总计** | ~700MB ✅ |

（机器总内存 1.6G，可用 ~934MB，压力可控）

---

## 阶段 ③：注入故障 + 通过监控发现

**目标**：主动触发 bug，在 Prometheus 里看到异常信号

**做什么**：
- 写压测脚本，触发 500 错误和慢响应
- 在 Prometheus 里观察：
  - 错误率上升（HTTP 5xx 计数）
  - 响应延迟 P99 飙升
  - 内存曲线持续走高
- 配置 Prometheus **告警规则**（Alert Rules）
  - 「5 分钟内错误率 > 10%」→ 告警
  - 「P99 延迟 > 2s」→ 告警
- 模拟收到「告警通知」，开始排查

---

## 阶段 ④：根因定位（RCA）

**目标**：从告警出发，用日志 + 指标定位到具体代码行

**做什么**：
- 从 Prometheus 告警 → 确定出问题的时间和接口
- 通过 `requestId` 在结构化日志中找到对应请求的完整调用链
- 结合代码，定位根因（是哪个参数、哪个分支导致的）
- 总结排查路径：
  ```
  告警 → 异常指标（哪个接口、什么时间段）
       → 查日志（requestId 关联）
       → 定位代码（具体行）
       → 修复
  ```

---

## 阶段 ⑤：改进闭环 — RCA 报告 + 补测试 + 加监控

**目标**：让这个问题 **不会再漏到线上**（这是测开的核心价值）

**做什么**：

**❶ 编写 RCA 报告**（模板）
```
- 故障时间线（发现→响应→定位→修复）
- 根因（代码层面的具体原因）
- 影响范围
- 为什么测试没拦住？
- 改进措施
```

**❷ 补充测试用例**
- 把导致故障的参数组合写进自动化测试
- 边界条件覆盖

**❸ 监控改进**
- 为这个 bug 类型添加专项告警规则
- 补充业务级别的监控指标（不只看 HTTP 状态码）

**❹ 总结方法论**
- 形成「线上问题 → 测试改进」的标准流程
