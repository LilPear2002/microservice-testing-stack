# 根因分析报告 (RCA Report)

> 报告编号：RCA-2026-001  
> 报告日期：2026-05-14  
> 作者：测试开发实习生  
> 严重程度：🟡 Medium（影响部分用户，非核心业务中断）

---

## 1. 故障概述

buggy-api 服务在压测期间出现 3 类异常：
- **异常 A**：`/api/report` 接口 P99 延迟飙升至 7.35s
- **异常 B**：`/api/process` 接口在传入非正数时频繁返回 500
- **异常 C**：`/api/cache` 接口导致服务内存持续增长

## 2. 故障时间线

| 时间 | 事件 |
|------|------|
| 13:15 | 开始对 buggy-api 进行常规压测（5并发，80%正常+20%异常流量） |
| 13:16 | Prometheus 检测到 `bug_process_zero_or_negative_total` 计数开始增长 |
| 13:16 | `HighLatency` 告警触发：P99 延迟突破 2s 阈值 |
| 13:17 | `BugProcessZeroValueFrequent` 告警触发：错误速率 0.052/s > 0.05/s |
| 13:18 | `MemoryLeakWarning` 进入 pending：缓存达 16661 bytes > 10000 |
| 13:18 | 压测结束，总请求 1016，成功率降至 65.8% |

**发现 → 响应耗时**：约 2 分钟（Prometheus 自动告警）

## 3. 根因分析

### 3.1 异常 A：HighLatency（P99 = 7.35s）

| 维度 | 详情 |
|------|------|
| **代码位置** | `app/main.py` L168-176 |
| **直接原因** | 30% 概率触发 `time.sleep(random.uniform(2.0, 8.0))` |
| **根本原因** | 模拟"数据库慢查询"的设计缺陷：同步阻塞 sleep，无超时控制 |
| **影响范围** | 30% 的 `/api/report` 请求受影响 |

### 3.2 异常 B：BugProcessZeroValueFrequent（间歇 500）

| 维度 | 详情 |
|------|------|
| **代码位置** | `app/main.py` L128-133 |
| **直接原因** | `value <= 0` 时执行 `100 / value`，value=0 触发 `ZeroDivisionError` |
| **根本原因** | **缺少输入参数校验**——未验证 value 必须 > 0 |
| **影响范围** | 所有传入 value≤0 的 `/api/process` 请求 |

### 3.3 异常 C：MemoryLeakWarning（内存泄漏）

| 维度 | 详情 |
|------|------|
| **代码位置** | `app/main.py` L228, L238 |
| **直接原因** | `_global_cache.append()` 每次调用追加数据，从不清理 |
| **根本原因** | 全局可变状态无淘汰策略（缺 LRU/TTL/容量限制） |
| **影响范围** | 持续调用最终导致 OOM |

## 4. 为什么测试没拦住？

| Bug | QA 漏出原因 | 改进方向 |
|-----|------------|---------|
| A: 慢接口 | 未覆盖**性能测试**场景 | 增加接口响应时间断言 |
| B: 间歇 500 | 未覆盖**边界值**测试（value=0, -1） | 补充分类等价类/边界值用例 |
| C: 内存泄漏 | 未覆盖**持续运行/压测**场景 | 增加 soak test（浸泡测试） |

## 5. 改进措施

### 5.1 代码修复

| Bug | 修复方案 | 优先级 |
|-----|---------|--------|
| 间歇 500 | 加参数校验 `if value <= 0: raise HTTPException(400)` | 🔴 P0 |
| 慢接口 | 加 `asyncio.timeout` 包装 + 降级返回 | 🟡 P1 |
| 内存泄漏 | 引入 `functools.lru_cache` 限制缓存大小 | 🟡 P1 |

### 5.2 测试补充

| 测试类型 | 覆盖场景 | 工具 |
|---------|---------|------|
| 边界值测试 | value = 0, -1, -100, 极小浮点数 | pytest |
| 性能测试 | `/api/report` P95 < 500ms | pytest-benchmark |
| 浸泡测试 | 连续调用 `/api/cache` 1000 次，内存不超限 | pytest |

### 5.3 监控改进

| 改进项 | 说明 |
|--------|------|
| 增加 P0 告警：接口可用性 | `/api/process` 错误率 > 50% → 立即通知 |
| 增加业务指标 | 正常/异常请求比例、慢请求占比趋势 |
| 增加内存告警阈值 | 进程 RSS > 200MB → critical |

## 6. 经验教训

1. **边界值测试不是可选的**——参数校验是安全第一道防线
2. **性能测试应该纳入 CI**——不只是功能正确，还要时间可控
3. **监控是测试的延伸**——线上发现的问题，必须反哺测试用例
4. **Bug 专项指标**能大幅加速问题定位（从"哪个接口爆了"直接到"哪个 Bug 爆了"）

---

> **结论**：3 个 Bug 的根因均为代码层缺陷，通过补充边界值测试 + 性能测试 + 浸泡测试可以在上线前拦截。监控告警在 2 分钟内完成了发现，表现良好。
